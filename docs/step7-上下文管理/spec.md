# Step 7 — 上下文管理

## 背景

CodePilot 当前的上下文管理仅基于最简单的滑动窗口策略（按固定轮数裁剪，默认保留最近 10 轮），存在明显短板：

- **无 token 感知**：窗口按「轮数」而非「token 体积」裁剪。单轮内巨大的工具结果（读取大文件、Grep 全仓、Bash 长输出）会把上下文撑爆，而裁剪却按轮数机械进行，无法在 token 层面精确控制。
- **无压缩能力**：逼近窗口上限时，唯一的保护是 AgentLoop 注入提示让模型「放弃并总结当前进展」——等于丢弃工作而非压缩保留，长任务无法持续。
- **工具结果是 token 大头却原样驻留**：ReadFile / Grep / Bash 的完整输出全文留在对话里，既浪费上下文，又拖累每次请求的成本与延迟。
- **无撞墙兜底**：Provider 返回 `prompt_too_long`（HTTP 400）时直接报错中断，用户最新输入可能因此丢失。

Step 7 的目标是给 CodePilot 装上「真正能用的」上下文管理能力，让它能在有限的 token 预算里长时间干活而不瘫掉。核心思路是**两层压缩策略**——轻量预防（管单条消息内工具结果体积）+ 重量兜底（管累积历史长度），保证对话累积再多也不会因上下文溢出而崩溃。

## 目标用户

- 进行长周期、多步骤编码任务的开发者（连续几十轮对话 + 大量工具调用）。
- 希望对话累积再多也不会因上下文溢出而中断或丢失工作的用户。
- 关心 token 成本与响应延迟（减少无谓的重复大块工具结果传输）的用户。

## 能力清单

1. **两层压缩策略**：第一层「轻量预防」管单条消息内工具结果体积；第二层「重量兜底」管累积历史长度，逼近窗口上限时整体摘要压缩。每次 API 请求前先跑第一层，再判断是否需要第二层。
2. **工具结果存盘与预览替换（第一层）**：单个工具结果超过阈值即落盘到会话目录下的工具结果归档目录，对话内存中仅保留「头部截断预览 + 文件路径提示」，并告知 LLM 需要准确结果时可用 ReadFile 重新读取；单条消息内多个工具结果合计超阈值时，按体积从大到小依次存盘替换。
3. **工具结果存盘子系统**：以工具调用 ID 为文件名、存放在对应会话目录下的专用子目录；写入幂等（文件已存在则跳过）；**保证 prompt cache 命中率**——某个工具结果一旦在某轮决定被替换为预览，则在后续整个会话中保持替换态（反之保持原文态），确保每次输入给 LLM 的 prompt 前缀稳定一致。
4. **用户原始消息原文保留**：压缩只对工具结果与历史对话动手，用户的原始输入消息尽量原文保留、不被摘要改写。
5. **结构化摘要压缩（第二层）**：整体对话逼近窗口上限时，调用 LLM 对较早的历史生成结构化摘要；近期原文保留（从尾部按 token 往回数，约 1 万 token 或至少 5 条消息）；摘要按固定部分组织。
6. **摘要 Prompt 约束**：明确禁止模型调用任何工具；要求模型先写分析草稿、再写正式摘要，草稿用完即丢、不进入最终摘要。
7. **压缩边界提示**：压缩完成后补一条边界消息，提示模型若需要文件细节请重新读取，不要照着摘要脑补代码。
8. **早期原文归档**：被摘要压缩掉的早期原文消息归档到会话目录下的历史归档文件，对话内存替换为「摘要 + 近期原文」的活跃视图；归档原文保留可供追溯。
9. **手动触发压缩**：用户可通过斜杠命令与 WebUI 状态栏按钮主动触发压缩。
10. **熔断保护**：摘要生成连续失败达到阈值即熔断，本会话停止自动压缩（避免死循环），但允许用户手动重试。
11. **Provider 撞墙兜底**：对 Provider 返回的上下文超长错误（`prompt_too_long`）有「紧急压缩 + 单次重试」机制，不让一次撞墙就丢掉用户最新输入。
12. **触发时机**：每次 API 请求前先跑轻量预防（管单条消息大小），再判断是否需要重量兜底（管累积历史长度）。
13. **安全余量分级**：自动触发时留较大安全余量（防 token 估算误差）；用户手动触发时余量收窄（用户主动要压，无需过大缓冲）。
14. **可配置性**：压缩相关阈值（工具结果存盘阈值、消息合计阈值、自动触发余量、近期原文保留量、熔断阈值等）均可通过配置文件覆盖默认值。
15. **可观测性（日志 + UI 感知）**：两层压缩的每次触发均有结构化日志（sessionID、触发层级、压缩前后 token、tool_use_id / 摘要条数、是否熔断）；UI 上第二层摘要压缩有明显用户感知（强提示，如 toast / 对话流标记节点），第一层轻量替换有轻量感知（状态栏计数/标记，不打扰）；熔断状态可见。

## 非功能要求

- **高性能**：压缩编排是每次 API 请求前的同步前置步骤，必须低开销——轻量预防以字符/token 估算为主，不引入额外 LLM 调用；仅第二层摘要与紧急压缩调用 LLM。存盘子系统使用幂等写入避免重复 IO。
- **高可用**：压缩任何环节失败不得中断主对话流程——存盘失败降级为保留原文；摘要失败计入熔断计数并降级；紧急压缩失败则保留原错误上报，不吞掉异常。
- **高扩展**：压缩策略以可插拔方式组织（轻量预防器 / 摘要压缩器 / 协调器职责分离），后续可替换或新增策略（如更精确的 tokenizer、图片内容压缩）而不改动调用方。
- **高安全**：工具结果存盘内容视为敏感数据（可能含代码 / 密钥 / 日志），存放在会话目录内、受现有路径沙箱约束；摘要 Prompt 严格禁止工具调用，防止压缩阶段触发非预期操作。
- **兼容性**：不破坏 Step 1~6 已有能力——滑动窗口作为兜底保留；旧会话（无压缩产物）在新代码下正常加载与渲染；append-only 持久化模型基本不被破坏（第二层摘要作为低频重组事件，单独走「重写活跃历史 + 追加归档」路径）。

## 设计骨架

代码组织于记忆层上下文管理包，与现有滑动窗口同层，符合「记忆层负责上下文管理」的架构约束：

```
src/internal/memory/context/
├── window.go              # 既有：滑动窗口（保留，作为兜底）
├── measure.go             # 新增：细粒度 token 度量（消息级 / block 级 / 累计）
├── tool_result_store.go   # 新增：工具结果存盘子系统（幂等写入 + 路径管理）
├── preview.go             # 新增：预览生成（头部截断 + 路径尾注）
├── light_compactor.go     # 新增：第一层轻量预防压缩器
├── summary_compactor.go   # 新增：第二层摘要压缩器（含摘要 Prompt）
├── compactor.go           # 新增：压缩协调器（编排两层 + 熔断 + 手动触发）
└── *_test.go              # 各组件单测

src/internal/memory/session/
└── archive.go             # 新增：历史原文归档 + 活跃历史重写能力
```

接入点：

- 引擎层 `runOneLLM` 在派生上下文视图（`GetContext`）前调用协调器跑压缩编排。
- 交互层 handler 提供 `/compact` 命令路由与 WebUI 按钮消息、压缩事件可观测性推送。
- 配置层 `setting.json` 新增压缩配置段。
- 主流程 `main.go` 顶层装配协调器并注入会话目录。

## Out of Scope（本步骤不做）

- **精确 tokenizer**（如 tiktoken / 按 cl100k_base 分桶的本地词表）：本步骤沿用现有字符级粗估 + API 返回的精确 `input_tokens`，精确 tokenizer 留待后续。
- **图片 / 多模态内容块**的压缩：本期仅处理文本与工具结果块。
- **跨会话的压缩状态持久化**（如把「替换状态记录」写盘）：压缩状态为会话内内存态，会话恢复后由确定性规则重新派生，不做跨会话状态存储。
- **完整的快捷命令系统**：`/compact` 仅作为临时接入点（命令路由最小实现），系统化的快捷命令框架属 Step 9。
- **自动记忆系统的深度集成**：本期压缩不向自动记忆写入，自动记忆召回仍走 Step 4 既有路径。
- **压缩质量的人工评分 / 反馈闭环**。

## 待确认语义点（实现时与用户定夺）

- 「手动触发时余量收窄到 3K」的精确语义：本 spec 采用的解读是——手动触发**不受当前剩余量限制**（用户主动随时可压），且压缩目标允许只预留 3K 余量（比自动的 13K 更激进）。若实际意图为「手动触发仅在剩余 ≤ 3K 时生效」等其它语义，需在实现前确认。
