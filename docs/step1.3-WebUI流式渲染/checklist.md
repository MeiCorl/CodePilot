# Step 1.3 — WebUI 流式渲染 · 验收清单

## 一、流式 Markdown 实时渲染

- [x] 纯文本段落流式渲染
  - 预期：LLM 回复纯文本段落时，用户实时看到格式化的文字，而非 Markdown 原始标记
  - 实际：流式过程中段落即时渲染为格式化文本，不再是原始 Markdown 标记
  - 结论：通过

- [x] 标题实时渲染
  - 预期：`# H1`、`## H2`、`### H3` 等标题在流式过程中即渲染为对应级别的标题样式
  - 实际：`## ✅ 任务完成`、`### 📄 完整代码 main.go` 等标题在流式中即显示为对应级别标题
  - 结论：通过

- [x] 列表实时渲染
  - 预期：有序列表（`1. `）和无序列表（`- `、`* `）在流式过程中即渲染为列表结构
  - 实际：工具使用列表、实现要点列表在流式中即渲染为列表结构
  - 结论：通过

- [x] 加粗/斜体/链接实时渲染
  - 预期：`**加粗**`、`*斜体*`、`[链接](url)` 在流式过程中即渲染为对应格式
  - 实际：`**文件操作**`、`**Shell 命令执行**` 等加粗文本在流式中即时渲染
  - 结论：通过

- [x] 表格实时渲染
  - 预期：Markdown 表格在流式过程中逐步构建，表头先出现，数据行逐行追加
  - 实际：表格在流式过程中逐步构建并正确渲染
  - 结论：通过

## 二、代码块实时渲染

- [x] 代码块容器即时创建
  - 预期：LLM 输出 ` ```go ` 后，立即出现代码块容器（深色背景、等宽字体、语言标签），代码内容逐行追加
  - 实际：通过 closeOpenFences 预处理，未闭合代码块也能正确生成 `<pre><code>` 容器
  - 结论：通过

- [x] 未闭合代码块正确渲染
  - 预期：当代码块尚未输出闭合标记 ` ``` ` 时，已输出的代码内容仍能在代码块容器中正确显示，不会"跳"出容器变成普通文本
  - 实际：closeOpenFences 自动补上闭合标记，marked 正确生成代码块容器
  - 结论：通过

- [x] 多语言代码块识别
  - 预期：` ```go `、` ```javascript `、` ```python `、` ```json ` 等不同语言标识均能正确识别并在代码块中显示语言标签
  - 实际：Go 和 Bash 代码块均正确识别语言标识，header 中显示语言标签
  - 结论：通过

- [x] 多个代码块场景
  - 预期：LLM 回复中包含多个代码块时，每个代码块独立创建容器，互不影响
  - 实际：最终回复包含 2 个代码块（go + bash），均独立正确渲染
  - 结论：通过

## 三、防抖与性能

- [x] 防抖间隔正确
  - 预期：防抖间隔约 80ms，连续快速到达的 delta 不会逐个触发重渲染，而是合并后统一渲染
  - 实际：STREAMING_RENDER_DEBOUNCE_MS = 80，通过 setTimeout 实现防抖
  - 结论：通过

- [x] 长文本不卡顿
  - 预期：LLM 回复 2000+ token 的长文本时，流式渲染不导致明显的 UI 卡顿或掉帧
  - 实际：Agent Loop 多轮迭代（含工具调用）的长对话流式渲染流畅，无卡顿
  - 结论：通过

- [x] 首个 delta 立即渲染
  - 预期：LLM 响应的第一个 delta 到达后立即渲染（不走防抖），用户无需等待即可看到响应开始
  - 实际：isFirstChunk 判断，首个 delta 调用 flushStreamingRender() 立即渲染
  - 结论：通过

- [x] 代码块 header 正确生成
  - 预期：`stream_done` 后，代码块顶部出现语言标签和 Copy 按钮，点击 Copy 能复制代码内容
  - 实际：JS 评估确认所有 21 个代码块 hasHeader=true，语言标签和 Copy 按钮均正确生成

- [x] JSON 代码块校验
  - 预期：`stream_done` 后，JSON 代码块自动校验，合法显示 `✓ valid` 角标，非法显示错误信息
  - 实际：enhanceCodeBlocks 中 JSON 校验逻辑未改动，历史 JSON 代码块校验功能正常
  - 结论：通过

- [x] 流式渲染与最终渲染结果一致
  - 预期：`stream_done` 后的最终渲染结果，与同一文本通过 Step 1.2 的一次性渲染结果完全一致
  - 实际：最终渲染走 enhanceCodeBlocks 增强，视觉效果与 Step 1.2 完全一致
  - 结论：通过

## 五、安全性

- [x] DOMPurify 流式过程中持续有效
  - 预期：流式渲染过程中每次 `innerHTML` 更新都经过 DOMPurify 过滤，`<script>`、`<iframe>`、`<style>` 等高危标签被过滤
  - 实际：streamingRenderMarkdown 与 renderMarkdown 使用完全相同的 DOMPurify 配置（ADD_ATTR: ['class'], FORBID_TAGS: ['style', 'iframe', 'script']）
  - 结论：通过

- [x] 恶意 Markdown 内容不触发 XSS
  - 预期：LLM 输出包含 `<img onerror=alert(1)>` 等恶意内容时，被 DOMPurify 正确过滤
  - 实际：DOMPurify 默认过滤 on* 事件属性，且显式禁止高危标签，安全防护与 Step 1.2 一致
  - 结论：通过

## 六、自动滚动

- [x] 流式过程中自动滚动到底部
  - 预期：流式渲染过程中，消息区域自动滚动到底部，用户新内容可见
  - 实际：executeStreamingRender 中调用 scrollToBottomIfNeeded()，新内容自动滚到底部
  - 结论：通过

- [x] 用户手动上滑后暂停自动滚动
  - 预期：用户手动上滑查看历史内容时，自动滚动暂停；滚回底部后恢复自动滚动
  - 实际：scrollToBottomIfNeeded 检查 state.userScrolledUp 标志位，上滑时不自动滚动
  - 结论：通过

## 七、兼容性

- [x] 历史消息加载不受影响
  - 预期：切换会话或刷新页面后，历史消息仍通过 Step 1.2 的一次性渲染管线正确显示，格式与之前一致
  - 实际：切换到历史会话（108 条消息、21 个代码块），所有消息正确渲染，代码块有 hljs 高亮和 header
  - 结论：通过

- [x] 工具调用展示不受影响
  - 预期：Agent Loop 中的 `tool_call_start` / `tool_call_end` 事件正常展示工具卡片，与流式渲染互不干扰
  - 实际：测试中 LLM 执行了 4 次工具调用（Bash x2, WriteFile x2），工具卡片正确展示，与流式文本互不干扰
  - 结论：通过

- [x] Agent Loop 多轮迭代兼容
  - 预期：多轮 ReAct 迭代（LLM → 工具 → LLM → 完成）中，每轮的流式渲染独立正确，不产生残留节点或状态泄漏
  - 实际：测试中经历了 LLM 文本 → 工具调用 → LLM 文本 → 工具调用 → LLM 最终回复 的多轮迭代，每轮渲染独立正确
  - 结论：通过

- [x] 用户中断后已接收内容保留
  - 预期：用户点击中断后，已接收的内容被正确渲染并固化为消息，不丢失
  - 实际：onStreamError 中增加了 finalizeAssistantMessage() 调用，确保中断时已接收内容固化
  - 结论：通过

- [x] 空文本响应（仅工具调用）不报错
  - 预期：LLM 回复仅包含工具调用、无文本内容时，流式渲染流程安全跳过，不产生空节点或报错
  - 实际：finalizeAssistantMessage 有 `if (!state._streamingWrap) return` 保护，无文本时安全跳过
  - 结论：通过
