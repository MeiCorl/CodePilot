# Auto Spec 工作流技能

被 `/auto-specs` 触发。本技能是 `specs` 的「全自动」版本：文档生成阶段与 `specs` 完全相同；**生成完三份 md 文档后会等用户做一次整体确认，确认后自动一口气跑完 `tasks.md` 中所有 task，task 之间无需用户介入**。

> 一句话区别：
> - **specs**：每完成一个 task → 停下来等用户确认
> - **auto-specs**：**文档生成后等一次用户确认** → 之后每完成一个 task → 自动派发下一个 task → 一口气跑完

**确认点只设置一处**：三份 md 文档刚写完的关卡。后续 task 推进过程中**不再询问**（仅 Agent 报告 blocked 时停下）。

---

## 核心架构：主会话编排 + Agent 派发

主会话作为编排者，**不亲自写代码**；每个 task 派发一个 `general-purpose` Agent（独立上下文，等价于 `/clear` 后的新会话）。Agent 通过 prompt 注入任务坐标 + 读取三份 md 文档获取全部必要信息，无需主会话的对话历史。

```
编排主循环:
  扫描 tasks.md → 找到状态非「已完成」的最小编号 task K
  → 派发 Agent(独立上下文)执行 task K
  → 等待 Agent 返回 JSON 报告
  → completed → 确认 tasks.md 状态 → 回到顶部
  → blocked   → 输出阻塞详情,等用户决策
  → 所有 task 完成 → 同步 PROGRESS.md → 结束
```

## 适用场景

| 场景 | 判定 | 后续动作 |
| --- | --- | --- |
| 新需求 | 用户提新功能想法,无对应 spec 文档 | 需求澄清 → 生成三份 md 文档 → 阶段 A 确认 → 编排主循环 |
| 断点续做 | 用户提到"回到第 N 步/继续上次的 XXX/调整之前的工作" | 读三份 md 文档 → 汇报当前状态 → 阶段 A 确认 → 编排主循环 |

**step 编号与 idea_name 规则**：扫描 `docs/step*` 已有目录 → 优先匹配 `CLAUDE.md` 预定义计划名表(1-12 步,见 PROGRESS.md 表格) → 已存在则复用,否则按编号创建 → 计划外的新功能由技能拟定编号,子任务用 `stepX.Y: ...` 形式

## 需求澄清

**核心原则**：每轮最多 3 个确认点；提问具体；不假设答案；关键点澄清后主动告知"需求已足够清晰,准备生成文档"。

**澄清要点(按优先级)**：

1. 核心问题：解决什么痛点?目标用户?
2. 能力边界：做什么 / 不做什么?
3. 交互方式：用户如何触发?
4. 边缘场景：异常输入怎么处理?
5. 与现有功能的关系：依赖什么 / 影响什么?
6. 非功能要求：性能/安全/兼容性?

## 三份文档规范

文档落盘到 `docs/{step_n-idea_name}/`,idea_name 优先使用 CLAUDE.md 预定义计划名(如 `docs/step2-工具系统集成/spec.md`)。

### spec.md — "做什么 / 不做 / 做到什么程度算完成"

**必须章节**：`# Step N — {功能名称}` + `## 背景` + `## 目标用户` + `## 能力清单` + `## 非功能要求` + `## 设计骨架`(目录结构示意) + `## Out of Scope`

**红线**：不写具体函数名/参数名/默认值/错误文本/行号/SDK 类型名(实现细节,易过期)

### tasks.md — "怎么做 / 按什么顺序 / 动什么文件"

- 3~10 个任务,每个能一次专注会话完成
- 每个任务标注: **状态**(`待完成`/`进行中`/`已完成`) / **目标** / **影响文件** / **依赖任务** / **具体内容**(详细到可直接开始实现) / **参考资料**(精确到函数名/行号/SDK 文档链接)
- **最后两个任务固定为**:`接入主流程` + `端到端验证`
- **状态管理规则**:开始实现前 → 改 `进行中`;完成后 → 改 `已完成`;每次变更实际编辑 tasks.md

**Task 模板**:
```markdown
## Task K: {任务名称}
**状态**: 待完成
**目标**: {一句话描述}
**影响文件**:
- `path/to/file.go` — 新建/修改, {说明}
**依赖**: Task X ({原因})
**具体内容**:
1. {步骤1}
2. {步骤2}
**参考资料**:
- {SDK 文档 / 函数签名 / 行号引用}
```

### checklist.md — "怎么验证 / 验收项是什么"

- 每项必须可勾选、可观测;不许"实现完整"这种模糊评价
- 包含「预期」「实际」「结论」三栏
- 把 spec 里被砍掉的具体值(错误文本/默认值/阈值)放进来作为验收项
- 至少一条端到端验收

**Item 模板**:
```markdown
- [ ] {功能点描述}
  - 预期: {具体的预期行为或结果}
  - 实际: {验证时填写}
  - 结论: {验证时填写 通过/不通过}
```

## 断点续做

1. 定位 `docs/{path}` 目录(用户说"第 N 步" → `docs/step{N}-*/`;说功能名 → 模糊匹配;找不到则列出已有步骤让用户选)
2. 顺序阅读 `spec.md` → `tasks.md` → `checklist.md`
3. 汇报:当前步骤 / 已完成 task / 下一个待办 task / checklist 通过情况
4. **直接进入阶段 A** 让用户做一次整体确认,然后自动跑

---

## 🚀 自动派发编排规则(与 specs 的核心差异)

### 阶段 A:文档生成完成 → 等用户确认 → 进入编排

1. **输出"文档已就绪"小结**:task 总数 + 入口目录路径 + 每份文档覆盖范围
2. **明确告诉用户这是 auto-specs 唯一需要确认的关卡**:
   > "回复「确认 / OK / 继续 / go」即开始派发 Agent 一口气跑完所有 task;如需调整文档请直接指出要改的章节/任务/验证项"
3. **等待用户回复**(流程中**唯一**的等待点)
4. **收到确认后**:标记"文档已确认,开始自动编排" → 进入编排模式 → 不再询问
5. **若用户要求调整文档**:按指示修改;改完**再次回到本阶段第 1 步**让用户重新确认(避免漏改)

### 阶段 B:编排主循环

```
loop:
  1. 扫描 tasks.md, 找到状态非「已完成」的最小编号 task K
  2. 若所有 task 完成 → break → 阶段 C
  3. 派发 Agent 执行 task K(详见阶段 D)
  4. 等待 Agent 返回
  5. 解析报告, 按 D.3 决策
  6. 回到顶部
```

**输出简洁日志**:`[Auto-Specs] → 派发 Agent 执行 Task K: {name}` / `[Auto-Specs] ← Agent 返回: completed (checklist X/Y 通过)`

### 阶段 C:整步收尾

1. 同步 `.harness/PROGRESS.md`(详见通用规则 5)
2. 输出"整步完成"总结:task 总数 / 最终 commit hash / 能力清单摘要
3. 自然终止

### 阶段 D:Agent 派发规范

**D.1 类型**:使用 `general-purpose` Agent(支持全套工具);**不要用** `Explore`(只读)或 `Plan`(只规划)

**D.2 Prompt 模板**(派发时替换 `{K}` `{task_name}` `{step_n-idea_name}` `{项目根}` 等占位符):

```text
你是 Auto-Specs 工作流中的 Task Worker。主会话已生成三份 md 文档,派发你独立完成第 {K} 个 task(独立上下文,不持有主会话对话历史)。

## 你的任务
- 工作目录: docs/{step_n-idea_name}/
- 目标 task: Task {K} ({task_name})
- 主项目根: {项目根绝对路径}

## 必读文档(按顺序)
1. {项目根}/docs/{step_n-idea_name}/spec.md — 全局需求与能力边界
2. {项目根}/docs/{step_n-idea_name}/tasks.md — 锁定 Task {K} 的具体内容/影响文件/参考资料
3. {项目根}/docs/{step_n-idea_name}/checklist.md — Task {K} 对应的验证项

## 工作流程
1. 校验 Task {K} 状态:仅当「待完成」或「进行中」时执行
2. 将 Task {K} 状态改为「进行中」(编辑 tasks.md)
3. 按 tasks.md 中 Task {K} 的「具体内容」逐步实现
4. 按 checklist.md 中 Task {K} 对应项逐项验证, 填入「实际」「结论」
5. **仅当所有验证项通过**时, 将 Task {K} 状态改为「已完成」; 否则保持「进行中」并说明未通过项
6. 在主项目根执行 `git add -A && git commit -m "Step {N} Task {K}: {task_name}"` (无 git 仓库可跳过)

## 返回报告(严格 JSON)
{
  "task": "{K}",
  "name": "{task_name}",
  "status": "completed" | "blocked",
  "files_changed": ["path/relative/to/repo/root", ...],
  "checklist": {
    "passed": <int>,
    "failed": <int>,
    "items": [{"desc": "验证项简述", "result": "pass" | "fail", "note": "若 fail 写原因"}]
  },
  "commit_hash": "<git commit hash, 无 git 时为 null>",
  "notes": "实现要点 / 遗留问题 / 阻塞原因(blocked 时必填)"
}

## 硬性约束
- 使用简体中文
- 严格遵循 .harness/PROGRESS.md、架构分层与系统设计规则
- SDK 使用务必通过 context7 MCP 查最新 API, 不凭记忆
- 不修改 Task {K} 范围以外的内容, 不修改 .harness/PROGRESS.md
- 失败/阻塞时不要硬撑, 明确报告 status=blocked
```

**D.3 主会话对返回值的处理**:

| Agent 状态 | checklist | 主会话动作 |
| --- | --- | --- |
| `completed` | 全部通过 | 确认 tasks.md 标为「已完成」, 派发下一个 task |
| `completed` | 有失败项 | 异常! 强制回退 tasks.md 为「进行中」, 重派同一个 task K 的修复 Agent |
| `blocked` | — | **停下编排**, 输出阻塞详情, 等用户决策(接受/调整/终止) |
| 调用本身失败 | — | 重试一次; 仍失败按 blocked 处理 |

### 阶段 E:异常与中断

| 场景 | 处理 |
| --- | --- |
| Agent 报告 blocked | 停下编排, 询问用户决策(编排阶段唯一会中断自动流程的场景) |
| 用户中途叫停(Ctrl+C) | tasks.md 持久化状态, 下次启动 `/auto-specs continue` 自动断点续做 |
| 用户要求改 spec/tasks/checklist | 停止编排, 按指示更新; **改完回到阶段 A 让用户重新确认**(防漏改) |
| Agent 谎报通过 | 主会话自动回退 + 重派(见 D.3) |

---

## 通用规则

1. **语言**:所有输出/提问/文档使用中文
2. **SDK 查询**:使用 SDK 时通过 context7 MCP 查最新 API, 不凭记忆
3. **文件路径**:`docs/{step_n-idea_name}/`,idea_name 优先 CLAUDE.md 预定义计划名
4. **实现过程**:
   - 文档生成完毕**必须等用户一次整体确认**才进入自动编排(阶段 A)——流程中**唯一**的等待点
   - 收到确认后, 主会话只做编排, **绝对不亲自写代码**
   - 每个 task 通过 Agent 派发实现上下文隔离
   - Agent 报告 `status=completed` 是唯一推进条件; `blocked` 必须停下等用户决策
5. **步骤完成后同步 PROGRESS.md**(由编排主循环在阶段 C 自动触发):更新 `📊 总览` + 追加 `✅ 已完成步骤` 章节(完成时间 `YYYY-MM-DD`、commit hash/message、文档链接、Task 数、能力列表 5-10 条) + 从 `🕓 待完成步骤` 删行 + 调整 `🧭 架构层覆盖度`;最终汇报中明确告知已更新
